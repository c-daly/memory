# Memory v0 — implementation plan (2026-05-10)

## Provenance

The original v0 plan dated 2026-05-09 was lost in the desktop spill — see `../2026-05-09-desktop-spill-losses.md` for the loss record. A re-derivation attempted on 2026-05-10 (`2026-05-10-memory-v0-rederived.md`) accumulated several inference errors that surfaced in conversation; that file is preserved as historical record. This plan is **not** a recovery of the lost original. It is a fresh plan reflecting current understanding as of 2026-05-10, which may or may not match what the original 2026-05-09 plan said.

## Premise (load-bearing facts)

From the four cross-cutting principles the memory narrative refers to (expressed across the 2026-05-07 and 2026-05-08 continuity decisions; not enumerated as a labeled list anywhere — articulated in conversation 2026-05-10):

1. **Sovereignty.** Each plugin is functional on its own. Install one without the rest; nothing required from siblings to operate.
2. **No cross-plugin coordination required at runtime.** Siblings can be absent or arrive later; absent siblings degrade gracefully.
3. **Provider structure.** Reader/Writer is the cross-plugin contract; semantics (schemas, paths, layout) live in the provider, not the consumer; multiple implementations per interface — implementation choice is configuration, not API.
4. **Plain text on disk.** The model is a valid reader without going through a provider. Provider APIs are convenience and consistency, not gatekeeping.

From `vault/10-projects/memory/narrative.md`:

- Memory is sovereign and self-contained.
- **Reader / Writer split.** `memory_reader` / `memory_writer` are the two entry points. Both bind to a configured `Provider`.
- **Four entry types** — `user`, `feedback`, `project`, `reference`. One file per entry with frontmatter `(name, description, type)` plus body. `MEMORY.md` is an index file (one line per entry) loaded on session start.
- Out of scope for v0: Obsidian extraction, cross-plugin coordination, continuity touch.

## Provider design — corrected from the re-derived plan

The re-derived plan made the `Provider` interface filesystem-shaped and the v0 reference impl a single `FilesystemProvider`. That was wrong on two counts:

1. **The Provider interface must accommodate non-filesystem implementations** even though none ship in v0. Future providers will wrap MCPs (Serena, memory itself reflexively, possibly others). The interface operations must be abstract enough that an MCP-backed impl is sensible — not "open file handle" or anything substrate-specific.
2. **Two providers ship in v0, not one.** `FilesystemProvider` (default; generic; no assumptions about Obsidian or any other plugin — works as a no-dependencies floor) AND `VaultProvider` (vault-aware; built on or composed with the filesystem provider; cheap to add given shared substrate). Both are filesystem-substrate; future MCP-substrate providers come later.

The "sample providers ship in `lib/providers/`" framing from the narrative was a packaging pragma — a way to ship vault-provider co-located with memory's code without committing to a separate `claude-providers` package or violating sovereignty. **Drop it as load-bearing on the design.** It's a shipping detail; the architecture is just "plugin tree contains its providers."

## Decisions to author during Phase 0

When the scaffold is created, write fresh dated decision records for these calls. The 2026-05-09 sample-providers decision was lost; this plan does **not** treat it as load-bearing, so it does not need re-deriving. Decisions to author:

- **Two providers in v0 (filesystem default + vault).** Why `FilesystemProvider` is the default; why `VaultProvider` ships alongside; why this is not the "vault provider out of scope" stance the narrative implied.
- **Provider interface accommodates non-filesystem impls.** Why operations are abstract rather than file-shaped; concrete implication: no file handles in the interface.
- **Four entry types, no schema registry.** v0 keeps it simple: hard-coded `user|feedback|project|reference`. Continuity's Phase 2 sketch imagined `mem.feedback`-style schemas; this plan deliberately doesn't follow that, per the narrative.

## Open questions

- Whether `memory_reader` exposes query (filter by name pattern, body grep) in v0. Recommend: read-by-name + list-by-type only; defer query to a Phase 3.x revision.
- `MEMORY.md` index format: confirmed mirror existing line-per-entry format (`- [name.md](name.md) — description`). All surviving dirs use this; harness already parses it.
- Whether `VaultProvider` uses the tenant path from `2026-05-08-reader-writer-architecture.md` (`<vault>/40-archive/memory/sources/*.md`) verbatim or makes the tenant configurable in v0. Recommend: use the documented path; configurability when a second deployment justifies it.
